# Auto‑Timing субтитров: подробный план реализации (RU)

Документ описывает **пошаговый план** реализации ключевой функции плагина CaptionPanels: **автоматически выставлять длительность (in/out) каждого блока субтитров** по реальному времени речи.

Основано на:
- `docs/AUTO_TIMING_PLAN.md`
- `docs/TRANSCRIBE_UTILITY_PLAN.md`

Статус: план/спецификация. Реализация будет по шагам, с обязательной документацией и Git‑историей.

---

## 1) Что считаем успехом (результат)

После импорта/создания субтитров у нас на таймлайне появляются слои‑блоки (например `Sub_VOICEOVER_2_3`). Сейчас они имеют фиксированную длительность шаблона.

Нужно, чтобы стало так:
- **Начало блока** = момент, когда диктор/спикер начинает говорить **первое слово блока**
- **Конец блока** = момент, когда заканчивается **последнее слово блока**
- Допускается, что блок «поедет» по таймлайну (мы двигаем и in, и out), главное — совпадение с речью.
- Работаем для **VOICEOVER** и **SYNCH**.
- После применения таймингов автоматически пересчитываем `subtitle_BG` (разрывы/группы).

Важно: импортируемый JSON (как «Хоккей») содержит длинные сегменты, а плагин режет их на много блоков. Поэтому авто‑тайминг должен работать **на уровне реальных блоков‑слоёв**, а не «сегментов исходного JSON».

---

## 2) Общая стратегия (двухфазная)

### Phase A (делаем первой): плагин применяет тайминги из файла

1) Плагин создаёт субтитры как сейчас.
2) Пользователь при необходимости вручную правит переносы/ошибки.
3) Плагин по кнопке **Export Blocks…** сохраняет `blocks.json` (список блоков: `segId + текст`).
4) Внешняя утилита (пока отдельно от плагина) делает транскрибацию и выравнивание и выдаёт `alignment.json`.
5) Плагин по кнопке **Apply Timings…** читает `alignment.json`, показывает отчёт «что изменится», и применяет in/out.

Плюсы: быстро проверить механику, минимум вопросов от ИБ (без распространения .exe внутри плагина).

### Phase U (параллельно/после): утилита транскрибации + выравнивания (WhisperX)

Утилита берёт `video.mp4 + blocks.json` и производит `alignment.json`.

Позже можно сделать Phase B: плагин запускает утилиту «в один клик» (но это отдельное согласование у ИБ).

## 3) Ключевое решение: стабильный идентификатор блока (segId)

Чтобы не «угадывать», какой тайминг к какому слою, каждый созданный блок должен иметь стабильный ID.

### Что такое segId в нашей системе

`segId` = **ID блока** (слоя субтитра), который реально существует на таймлайне.

Рекомендованная реализация:
- `segId` берём равным имени слоя на момент создания (оно у нас уникальное и предсказуемое):
  - `Sub_VOICEOVER_<batch>_<index>`
  - `Sub_SYNCH_<batch>_<index>`
- записываем его не только в name, но и в **layer.comment** (чтобы если кто‑то переименует слой, ID не потерялся):
  - `CP_SEGID=Sub_VOICEOVER_2_3`

Дополнительно (по желанию) можно дублировать в marker слоя, но на первом этапе достаточно comment.

## 4) Контракты данных (файлы и форматы)

### 4.1 blocks.json (выход плагина, вход утилиты)

Назначение: «передать утилите реальные блоки субтитров после правок».

Рекомендуемая структура:
```json
{
  "schemaVersion": 1,
  "source": {
    "app": "CaptionPanels",
    "uiVersion": "X.Y.Z",
    "compName": "...",
    "fps": 25
  },
  "blocks": [
    {
      "segId": "Sub_VOICEOVER_2_3",
      "type": "voiceover",
      "text": "Строка1
Строка2
Строка3",
      "textOneLine": "Строка1 Строка2 Строка3"
    }
  ]
}
```

Примечания:
- `text` — как видно пользователю (строки), `textOneLine` — удобно для матчинга.
- `segId` должен быть уникален внутри активной композиции.

### 4.2 alignment.json (выход утилиты, вход плагина)

Назначение: «для каждого segId дать start/end речи в секундах».

Структура (как в текущем плане):
```json
{
  "schemaVersion": 1,
  "source": {
    "engine": "whisperx|faster-whisper|other",
    "language": "ru",
    "media": "video.mp4"
  },
  "settings": {
    "padStartFrames": 0,
    "padEndFrames": 2,
    "minDurationFrames": 3
  },
  "blocks": [
    { "segId": "Sub_VOICEOVER_2_3", "start": 12.345, "end": 15.678, "confidence": 0.92 }
  ],
  "unmatched": [
    { "segId": "Sub_SYNCH_4_1", "reason": "foreign_speech|low_confidence|not_found" }
  ]
}
```

Примечания:
- `start/end` — секунды.
- `confidence` полезен для отчёта и отладки.

### 4.3 words.json (debug)

Не обязателен для плагина. Нужен, чтобы разбирать сложные кейсы выравнивания.

## 5) Реализация Phase A (плагин) — пошагово

### A0. Подготовка ветки и папок

Git:
- создаём ветку: `feature/auto-timing`
- базу выбираем так:
  - если авто‑тайминг зависит от текущих настроек/конфига из `feature/word-import` — **ветку делаем от `feature/word-import`**.

Документация:
- этот документ фиксирует общий план.
- дополнительно создадим позже:
  - `docs/AUTO_TIMING_WORKFLOW.md` (инструкция пользователю)
  - `docs/AUTO_TIMING_SECURITY_NOTES.md` (для ИБ)
  - `docs/schemas/blocks.schema.json` и `docs/schemas/alignment.schema.json` (опционально)

Коммит‑правило:
- каждый шаг ниже — отдельный коммит или 2–3 коммита (код + доки).

---

### A1. Проставление segId при создании субтитров

Цель: каждый новый `Sub_*` слой получает `CP_SEGID` в `layer.comment`.

Где:
- `cep_src/ContentPanels/host/lib/subtitles.jsx`

Что делаем:
1) В месте, где создаётся `newL`:
   - формируем `segId = newL.name` (после присвоения имени)
   - пишем `newL.comment = "CP_SEGID=" + segId` (или добавляем к существующему комменту аккуратно)
2) Добавляем утилитарную функцию «починить старые проекты»:
   - пройти по слоям активной композиции
   - найти `Sub_VOICEOVER_*` и `Sub_SYNCH_*`
   - если в comment нет `CP_SEGID=...` — записать

Почему важно:
- в реальной жизни слои могут переименовать. Comment остаётся якорем.

---

### A2. Export Blocks… (формирование blocks.json)

Цель: дать утилите точные данные о блоках после ручных правок.

UI:
- кнопка `Export Blocks…` в разделе субтитров.

Host‑логика:
1) Сканируем активную композицию.
2) Находим все слои субтитров (по имени `Sub_VOICEOVER_` / `Sub_SYNCH_`).
3) Для каждого слоя берём:
   - `segId` из `CP_SEGID` (если нет — fallback `layer.name`)
   - `type` (voiceover/synch)
   - `text` из Source Text (учитываем ``), делаем `textOneLine`
4) Сохраняем `blocks.json` в папку, доступную на запись.

Пути:
- путь задаём через `config.json`, чтобы потом менять без пересборки.
- рекомендуемый дефолт для Windows: `%LOCALAPPDATA%\CaptionPanels\AutoTiming\blocks\` (или `%TEMP%...`).

Отчёт пользователю:
- после экспорта показать, куда файл сохранился.

---

### A3. Apply Timings… (загрузка alignment.json и предварительный отчёт)

UI:
- кнопка `Apply Timings…`
- выбор файла `alignment.json`

Host‑логика (предпросмотр):
1) Прочитать `alignment.json`.
2) Сопоставить `segId` -> слой в активной композиции.
3) Для каждого блока посчитать:
   - old in/out
   - new in/out (ещё не применяем!)
   - итоговые кадры/секунды
   - предупреждения: слой не найден, end < start, confidence низкий, etc.
4) Вернуть в UI массив строк отчёта.

Критически важный момент: привязка секунд к таймлайну композиции

Проблема: `start/end` в alignment — «секунды от начала видео», а в AE субтитры могут начинаться не с 0.

Решение (якорь):
- выбираем **anchor block** — самый ранний по времени слой, который найден в alignment.
- берём:
  - `anchorLayerIn = layer.inPoint` (в секундах AE)
  - `anchorStartSec = alignment.blocks[anchorSegId].start`
- для любого блока:
  - `newInSec  = anchorLayerIn + (startSec - anchorStartSec)`
  - `newOutSec = anchorLayerIn + (endSec   - anchorStartSec)`

Дальше применяем правила:
- seconds -> frames (floor/ceil)
- padStartFrames / padEndFrames
- minDurationFrames
- clamp по длительности композиции
- (опционально) анти‑overlap — решим после тестов

---

### A4. Окно отчёта + Apply all / Apply except selected

Поведение:
1) После `Apply Timings…` показываем таблицу «Было / Стало»:
   - segId
   - old start/end
   - new start/end
   - confidence
   - checkbox Exclude
2) Кнопки:
   - `Apply all`
   - `Apply all except selected`

Host‑логика применения:
1) Получаем список excluded segId.
2) Применяем in/out только к разрешённым.
3) После применения:
   - пересчитываем `subtitle_BG` (обязательно)
   - возвращаем итоговый summary: сколько применено/пропущено/ошибок

---

### A5. Автоматический пересчёт subtitle_BG

После применения таймингов блоки могут «разъехаться» и появятся разрывы.

Правило:
- после успешного применения таймингов вызываем текущий механизм `_updateSubtitleBg(comp)`.

---

### A6. Логи и диагностика

Чтобы удобно разбирать ошибки:
- каждый запуск Apply Timings пишет лог (куда уже пишем остальные логи CaptionPanels).
- в UI показываем короткую ошибку + «путь к логу».

Минимум в логе:
- выбранный файл
- anchor segId
- сколько блоков найдено / не найдено
- список unmatched

---

### A7. Тестирование Phase A (без утилиты)

Тестируем по ступеням:
1) Создаём субтитры в реальной композиции.
2) Ручная правка пары блоков.
3) Export Blocks… -> проверяем blocks.json.
4) Делаем вручную маленький alignment.json на 3–5 segId.
5) Apply Timings… -> смотрим отчёт.
6) Применяем -> проверяем:
   - блоки двигаются и меняют длительность
   - `subtitle_BG` корректно «рвётся» на разрывах

После этого Phase A считается готовой для подключения утилиты.

## 6) Реализация утилиты (Phase U) — краткий план разработки

Подробности есть в `docs/TRANSCRIBE_UTILITY_PLAN.md`. Здесь — «как будем действовать».

### U1. Прототип (Python + WhisperX) на личном Windows ноутбуке

Цель: добиться корректного `alignment.json` на реальных роликах.

Шаги:
1) mp4 -> wav (ffmpeg)
2) ASR (Whisper)
3) WhisperX alignment -> words timestamps
4) matching blocks.json -> words -> blocks timings
5) generate alignment.json + words.json + лог

### U2. Packaging “portable folder” (для распространения)

Требования:
- зафиксированные версии зависимостей
- onedir (папка с exe + libs)
- модели отдельно (локальный кэш + опционально SMB‑источник)
- понятные ошибки, лог, `--version`

### U3. Интеграция one‑click (позже, после согласования ИБ)

Плагин:
- экспортирует blocks.json
- запускает утилиту
- применяет alignment.json
