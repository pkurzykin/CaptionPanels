# Дорожная карта развития архитектуры CaptionPanels (RU)

Дата: 2026-02-09
Ветка/состояние: текущая рабочая схема — WebView2 UI + bridge + JSX (ExtendScript) + внешние утилиты (word2json, WhisperX/align).

## Ключевая мысль
Сейчас архитектура **нормальная для AE‑плагинов**, но «костыльность» ощущается из‑за:
- строкового RPC (evalScript/bridge), слабой типизации и сложной диагностики;
- тяжелых операций внутри AE/JSX (риск подвисаний);
- разрозненных путей/конфигов/артефактов.

Цель — оставить UI на WebView2 (это ок), но сделать систему **предсказуемой, наблюдаемой и масштабируемой**.

---

## Этап 0 (уже сделано частично)
**Стабильные внешние утилиты + единый data-root**
- `C:\CaptionPanelsLocal\CaptionPanelsData\...` для артефактов, логов и кэшей.
- Внешние инструменты как portable‑папки (word2json, whisperx env) в `C:\CaptionPanelsLocal\CaptionPanelTools\...`.
- Техническое правило: не собирать сложные inline-команды `cmd /C` с вложенными кавычками; запуск внешних утилит делать через временный `.cmd` (устраняет повторяющиеся ошибки `C:\\Program`, `не является внутренней или внешней командой`).

Результат: минимальная зависимость от Program Files/UNC и понятная структура для ИБ.

## Этап 1 — «Довести текущую схему до взрослого уровня» (без переписывания UI)
Срок: 1–2 недели итерациями (параллельно фичам).

### 1.1. Единый протокол сообщений UI <-> Host
**Что делаем**
- Вводим единый формат ответа/ошибки для всех вызовов:
  - `ok`, `error`, `result`, `requestId`, `ts`, `module`, `fn`.
- Таймауты на UI‑стороне + понятные сообщения.
- Единый helper: `callHost(fnName, payload)` вместо «ручной сборки строк».

**Зачем**
- меньше «магии строк», проще дебажить, меньше “тихих” поломок.

Статус: **DONE (UI layer)** — добавлен helper `callHost(...)` с `requestId/ts/module/fn`, все рабочие UI-модули переведены с прямых вызовов на единый wrapper.

### 1.2. Нормальная диагностика (observability)
**Что делаем**
- Логи:
  - UI‑лог (в память + опционально в файл через host)
  - host‑лог
  - логи внешних утилит
- «Папка запуска» (runId) для каждой операции: `blocks/`, `whisperx/`, `alignment/`, `logs/`.
- Кнопка/экран «Diagnostics» (минимальный): показать последние runId, пути, ошибки.

**Зачем**
- любой баг превращается из «не работает» в «вот файл/команда/ошибка».

Статус: **DONE** (добавлен Diagnostics-экран в UI: snapshot путей/логов + последние `requestId` вызовов).

### 1.3. Упорядочивание конфигурации
**Что делаем**
- `config.json` делим логически:
  - `paths` (data/tools)
  - `asr` (whisper параметры)
  - `subtitle` (charsPerLine, bgGapSec)
  - `speakers` (dbPath)
- В Settings — только самое нужное, остальное оставить в конфиге.

**Артефакты/готовность**
- Док: `docs/CONFIG_REFERENCE.md` (ключи, дефолты, примеры).
- Док: `docs/TROUBLESHOOTING.md` (куда смотреть при ошибках).

**Git**
- Работа через ветки `feature/*`.
- Каждый этап — отдельные коммиты с понятными сообщениями.

Статус: **DONE** (шиппед-конфиг переведен на секции `paths/asr/subtitle/speakers`, runtime сохраняет совместимость со старыми flat-ключами).

## Этап 2 — «Тяжелое — наружу, AE — только применяет результат»
Срок: 2–4 недели (основной выигрыш по стабильности/производительности).

### 2.1. Pipeline‑модель (job/runs)
**Что делаем**
- Любая большая операция оформляется как “job”:
  - входные данные (пути, параметры)
  - ожидаемые выходные файлы
  - runId
- AE/JSX делает минимум:
  - экспорт блоков
  - применение таймингов
  - построение/обновление BG

Статус: **IN PROGRESS** (добавлен run-registry и `run.json` манифесты для `word_import`/`auto_timing`, вывод последних run в Diagnostics).

### 2.2. Вынос вычислений
**Что делаем**
- WhisperX/align — уже внешние.
- Следующий шаг: улучшить aligner и сделать “re-run alignment” без повторного ASR.
- Возможный будущий вынос: часть логики типографики/разметки (если понадобится).

### 2.2.5. FFmpeg как portable‑утилита (без PATH)
**Проблема**
- На рабочих ПК ffmpeg может отсутствовать.
- Править системный `PATH` часто запрещено (ИБ/политики).

**Что делаем**
- Держим ffmpeg как часть нашего "tools"‑набора:
  - `C:\CaptionPanelsLocal\CaptionPanelTools\ffmpeg\ffmpeg.exe`
- Вводим конфиг‑ключ (и используем его везде, где нужен ffmpeg):
  - `ffmpegExePath`: `C:/CaptionPanelsLocal/CaptionPanelTools/ffmpeg/ffmpeg.exe`
- Любые вызовы ffmpeg делаем по **явному пути**, без зависимости от `PATH`.

**Результат**
- Предсказуемый деплой: копируем папку tools — и всё работает.
- Проще согласование: один список внешних бинарников в одном месте.

### 2.3. Офлайн‑режим как стандарт
**Что делаем**
- Все модели/кэши держим локально или в согласованной общей папке.
- Включаем режим “no internet required”: если нет модели — выдаем инструкцию “куда положить”.

**Артефакты/готовность**
- Док: `docs/DEPLOYMENT_GUIDE.md` (как поставить на рабочий ПК: папки, права, пути).
- Док для ИБ: список исполняемых файлов и что они делают (без “черных ящиков”).

## Метрики “стало лучше”
- Меньше подвисаний AE при больших проектах.
- Повторяемость: один и тот же вход -> один и тот же результат.
- Диагностика: любая ошибка имеет лог + runId + команду.
- Установка на новый ПК занимает < 15 минут по инструкции.
